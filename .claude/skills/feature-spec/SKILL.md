# 기능 기획 구체화 (Feature Spec)

**이 스킬의 최종 산출물은 기획안 문서다.** 코드가 아니다. 스킬의 성공 여부는 "바로 구현에 착수할 수 있는 수준의 기능 기획안이 나왔는가"로만 판단하며, 구현 여부는 기획안 확정 후 사용자가 별도로 결정한다.
핵심 수단은 "지속적인 질문"이다. 단, 무작정 많이 묻는 게 아니라 **단계별로, 그 단계의 의사결정에 필요한 질문만** 던진다.

## 절대 원칙

1. **코드부터 쓰지 않는다.** 이 스킬이 발동되면 기획 문서가 완성되고 사용자가 확인하기 전까지 구현을 시작하지 않는다.
2. **질문은 AskUserQuestion 도구로 한다.** 각 라운드에 질문 최대 4개. 반드시 구체적인 선택지를 제시하고, 추천안이 있으면 첫 번째 옵션에 "(추천)"을 붙인다.
3. **한 단계의 결정이 끝나야 다음 단계로 넘어간다.** 단계를 건너뛰거나 여러 단계 질문을 섞지 않는다.
4. **이미 답이 나온 것은 다시 묻지 않는다.** 사용자의 최초 요청, 대화 맥락, 기존 코드베이스, 기존 기획 문서(`docs/planning/`)에서 파악 가능한 내용은 질문 대신 조사하고, 파악한 내용을 "이렇게 이해했다"고 확인만 받는다.
5. **사용자가 "모르겠다 / 알아서 해줘"를 선택하면** 근거 있는 추천안을 채택하고, 문서에 `[가정]` 표시로 기록한다. 가정이 쌓이면 마지막에 한꺼번에 재확인한다.
6. **단계가 끝날 때마다 문서에 즉시 기록한다.** 대화가 길어져도 결정사항이 유실되지 않도록, 각 단계 완료 시 산출물 문서를 갱신 저장한다.

## 진행 단계

### 0단계 — 착수

- 사용자의 최초 요청을 한 문단으로 재서술하고, 기존 코드/문서를 훑어 이미 결정된 것을 정리한다.
- 산출물 파일을 생성한다: `docs/planning/<기능명>-기능기획.md` (기능명은 kebab-case 한글/영문 허용)
- "지금부터 N단계에 걸쳐 질문하며 기획을 구체화한다"고 사용자에게 안내한다.

### 1단계 — 문제 정의와 목표

결정할 것: 이 기능은 **누구의 어떤 문제**를 풀며, **무엇이 되면 성공**인가.

질문 예시:
- 주 타깃 사용자는 누구인가? (선택지: 구체적 페르소나 후보들)
- 이 기능이 없을 때 사용자는 지금 어떻게 하고 있나?
- 성공을 무엇으로 판단하나? (사용률, 전환, 재방문 등 측정 가능한 기준)
- 왜 지금 만드나? (우선순위 근거)

### 2단계 — 사용자 시나리오

결정할 것: 대표 사용 시나리오(유스케이스) 2~4개와 각각의 시작-끝.

- 각 시나리오를 "누가 / 어떤 상황에서 / 무엇을 해서 / 무엇을 얻는다" 형식으로 작성해 확인받는다.
- 주 시나리오와 부 시나리오를 구분한다.
- 비로그인/신규/재방문 등 사용자 상태별 차이가 있는지 확인한다.

### 3단계 — 범위 결정 (스코프)

결정할 것: 이번에 만드는 것과 **만들지 않는 것**.

- 기능 후보 목록을 도출한 뒤 MoSCoW(Must/Should/Could/Won't)로 분류하는 질문을 던진다.
- **비범위(Won't) 명시가 가장 중요하다.** "이번엔 안 만든다"를 사용자 입으로 결정하게 한다.
- MVP 기준: Must만으로 1단계 시나리오가 끝까지 돌아가는지 검증한다. 안 돌아가면 범위를 다시 묻는다.

### 4단계 — 정책과 규칙 상세화

결정할 것: Must/Should 기능 각각의 **구현자가 헷갈릴 지점**.

기능마다 다음 관점으로 빠진 결정을 찾아 질문한다:
- **상태와 전이**: 이 대상이 가질 수 있는 상태는? 언제 바뀌나? (예: 펀딩의 진행중/달성/마감)
- **경계값과 한도**: 최소/최대, 글자 수, 금액, 기간, 개수 제한
- **권한**: 누가 보고, 누가 수정하고, 누가 삭제하나
- **엣지케이스**: 동시에 일어나면? 중간에 취소하면? 대상이 삭제된 뒤 접근하면? 중복이면?
- **데이터 정책**: 언제까지 보관? 삭제는 soft/hard? 개인정보 포함 여부
- 한 라운드에 다 못 물으면 기능별로 라운드를 나눈다. 사소한 것은 `[가정]`으로 채우고 넘어간다.

### 5단계 — 데이터와 시스템 관점

결정할 것: 구현 관점의 뼈대.

- 핵심 엔티티와 관계를 도출해 확인받는다 (표 또는 mermaid ER 다이어그램).
- 외부 연동(결제, 알림, 인증 등) 필요 여부와 방식 — **새 라이브러리/서비스 도입은 반드시 사용자 승인을 받는다.**
- 비기능 요구: 예상 트래픽 규모, 성능/보안상 특별히 신경 쓸 것이 있는지.

### 6단계 — 마무리와 검증

- 쌓인 `[가정]`을 모두 나열해 한 번에 확인받는다.
- 마일스톤(구현 순서)을 제안하고 확정한다.
- 완성된 문서 요약을 보여주고, 미해결 항목은 "오픈 이슈" 섹션에 남긴다.
- UI/UX 설계가 필요한 기능이면 `/ux-spec` 진행을 제안한다.

## 산출물 문서 템플릿

```markdown
# <기능명> 기능 기획

- 작성일: YYYY-MM-DD / 상태: 초안 | 확정
- 연관 문서: (UI/UX 기획 등)

## 1. 문제 정의
- 타깃 사용자 / 해결하는 문제 / 현재의 대안 / 성공 기준

## 2. 사용자 시나리오
- 주 시나리오, 부 시나리오 (누가/상황/행동/결과)

## 3. 범위
- Must / Should / Could / **Won't (이번에 안 만드는 것)**

## 4. 상세 정책
기능별로:
- 동작 규칙 / 상태 정의 / 경계값 / 권한 / 엣지케이스 처리
- `[가정]` 표시된 항목은 사용자 미확정 사항

## 5. 데이터 및 시스템
- 엔티티와 관계 / 외부 연동 / 비기능 요구

## 6. 마일스톤
- 구현 순서와 각 단계의 완료 기준

## 7. 오픈 이슈
- 미결정 사항과 결정 기한/조건
```

## 완료 기준

다음을 모두 만족하면 스킬 종료:
- [ ] Won't(비범위)가 명시돼 있다
- [ ] Must 기능 각각에 상태/경계값/권한/엣지케이스 결정이 있다 (또는 `[가정]`으로 표기)
- [ ] 확인 안 받은 `[가정]`이 없다
- [ ] 사용자가 최종 문서를 확인했다
