# Server 설계 기획 (Server Spec)

**이 스킬의 최종 산출물은 아키텍처와 DB 수준의 server 설계 기획안 문서다.** 코드가 아니다. 구현 여부는 기획안 확정 후 사용자가 별도로 결정한다.

기능 기획(/feature-spec)이 "무엇을 만드나", UX 기획(/ux-spec)이 "사용자가 어떻게 보나"를 답한다면, 이 스킬은 **"기술적으로 어떻게 구성하나"**를 답한다.

## 앞의 두 skill과 다른 점: 제안 주도로 진행한다

기능/UX 기획은 사용자만 답할 수 있는 제품 결정이라 질문 주도였다. server 설계는 기획안이 확정되어 있으면 대부분 엔지니어가 도출할 수 있다. 따라서:

- **내(Claude)가 설계안을 만들어 제시하고 확인받는 것**이 기본이다. 사용자에게 백지 질문을 던지지 않는다.
- AskUserQuestion은 **진짜 트레이드오프가 있는 지점에만** 쓴다 (예: 폴링 vs SSE, 정규화 vs 비정규화, 즉시 삭제 vs 배치 삭제). 선택지마다 장단점과 추천안을 명시한다.
- 정답이 명확한 것(네이밍, 인덱스, 응답 포맷 등)은 묻지 말고 설계안에 반영한 뒤 문서 리뷰에서 한꺼번에 확인받는다.

## 절대 원칙

1. **선행 문서를 먼저 읽는다.** `docs/planning/`의 기능 기획안(필수)과 UX 기획안(있으면)을 읽고 시작한다. 기능 기획안이 없으면 `/feature-spec`을 먼저 진행하자고 제안한다.
2. **기존 코드베이스가 곧 제약이다.** 현재 스택, 기존 스키마, 기존 API 패턴을 조사하고 그 관례를 따른다. **새 라이브러리·인프라·외부 서비스 도입은 반드시 사용자 승인을 받는다.**
3. **규모에 맞게 설계한다.** 기획안에 적힌 예상 규모를 넘는 과잉 설계(불필요한 캐시 레이어, 마이크로서비스 분리, 이벤트 큐 등)를 하지 않는다. 확장 여지는 "오픈 이슈"에 메모로만 남긴다.
4. **기획안의 정책을 빠짐없이 스키마와 API에 반영한다.** 기능 기획안의 상태 정의·경계값·권한·데이터 보존 정책 하나하나가 설계 어디에 반영됐는지 추적 가능해야 한다. 반영 불가능하거나 모순인 정책을 발견하면 설계를 임의로 바꾸지 말고 사용자에게 보고한다.
5. **단계가 끝날 때마다 산출물 문서를 즉시 갱신 저장한다.** 미확정 결정은 `[가정]`으로 표기하고 마지막에 일괄 확인받는다.

## 진행 단계

### 0단계 — 착수

- 기능 기획안·UX 기획안을 읽고, 설계에 영향 주는 확정 사항(정책, 규모, 권한)을 요약한다.
- 코드베이스를 조사한다: 스택과 버전, 기존 DB 스키마, 기존 API 라우트 패턴, 인증 방식.
- 산출물 파일 생성: `docs/planning/<기능명>-server기획.md`

### 1단계 — 아키텍처 구성

결정할 것: 시스템 전체 그림.

- 구성 요소와 데이터 흐름을 mermaid 다이어그램으로 제시한다 (클라이언트 ↔ 서버 ↔ DB ↔ 외부 서비스).
- 기존 아키텍처 관례를 기본값으로 하고, 벗어나야 할 이유가 있을 때만 트레이드오프 질문을 던진다.
- 결정 대상 예시: 실시간성 구현 방식(폴링/SSE/웹소켓), 비동기 처리 필요 여부, 외부 서비스 연동 방식, 파일/이미지 저장 위치.

### 2단계 — 데이터 모델 (DB 설계)

결정할 것: 스키마 전체.

- 엔티티 관계도(mermaid ERD)와 테이블별 정의를 제시한다: 필드, 타입, 제약(unique/nullable/default), 관계, 인덱스.
- 기획안의 정책을 스키마로 번역한다: 상태 정의 → enum/상태 컬럼, 경계값 → 제약, 보존 정책 → soft/hard delete 컬럼 설계.
- 기존 스키마와의 관계를 명시한다: 새 테이블인지, 기존 테이블 변경인지, 마이그레이션 영향(기존 데이터 처리 포함).
- 트레이드오프가 있는 지점(비정규화, 집계 컬럼 vs 매번 계산 등)만 질문한다.

### 3단계 — API 설계

결정할 것: 클라이언트-서버 계약.

- 엔드포인트 목록을 표로 제시한다: 메서드/경로/목적/권한/주요 파라미터.
- 핵심 엔드포인트는 요청·응답 예시와 에러 응답(상태 코드, 에러 케이스)까지 정의한다.
- UX 기획안의 화면·플로우와 대조해 빠진 API가 없는지 검증한다 (화면별 필요 데이터 ↔ 엔드포인트 매핑).
- 기존 API 패턴(라우팅 방식, 응답 포맷, 에러 관례)을 따른다.

### 4단계 — 횡단 관심사

결정할 것: 모든 API에 공통 적용되는 규칙.

- 인증·권한: 각 엔드포인트의 접근 제어를 기획안의 권한 정책과 대조한다.
- 동시성·정합성: 동시 요청 시 깨질 수 있는 지점(중복 생성, 잔액/수량 경합 등)과 처리 방식(트랜잭션, unique 제약, 낙관적 잠금).
- 입력 검증: 경계값 정책의 검증 위치(클라이언트/서버 이중 검증 원칙).
- 에러 처리와 실패 복구: 외부 연동 실패 시 동작.

### 5단계 — 마무리와 검증

- **정책 추적표**를 만든다: 기능 기획안의 각 정책 → 반영된 설계 위치(테이블/컬럼/API). 누락이 있으면 해당 단계로 돌아간다.
- 쌓인 `[가정]`을 일괄 확인받는다.
- 구현 순서(마이그레이션 → API → 연동)를 제안하고, 미결정 사항은 "오픈 이슈"에 남긴다.
- 문서 요약을 보여주고 사용자 확인으로 종료한다.

```markdown
# <기능명> Server 설계 기획

- 작성일: YYYY-MM-DD / 상태: 초안 | 확정
- 연관 문서: <기능명>-기능기획.md, <기능명>-uiux기획.md

## 1. 아키텍처
- 구성 다이어그램 (mermaid) / 주요 기술 결정과 근거

## 2. 데이터 모델
- ERD (mermaid) / 테이블별 정의 (필드·타입·제약·인덱스)
- 마이그레이션 영향 (기존 데이터 처리 포함)

## 3. API 설계
- 엔드포인트 목록 (메서드/경로/권한)
- 핵심 API의 요청·응답·에러 정의

## 4. 횡단 관심사
- 인증·권한 / 동시성 처리 / 입력 검증 / 에러·실패 처리

## 5. 정책 추적표
- 기능 기획안 정책 ↔ 설계 반영 위치 매핑

## 6. 구현 순서 및 오픈 이슈
- `[가정]` 표시된 항목은 사용자 미확정 사항
```

## 완료 기준

- [ ] 기능 기획안의 모든 정책이 정책 추적표에서 설계 위치와 매핑된다
- [ ] UX 기획안의 모든 화면이 필요한 API를 갖는다 (UX 기획안이 있는 경우)
- [ ] 스키마에 상태·제약·인덱스·마이그레이션 영향이 정의돼 있다
- [ ] 동시성으로 깨질 수 있는 지점과 처리 방식이 명시돼 있다
- [ ] 확인 안 받은 `[가정]`이 없다
- [ ] 사용자가 최종 문서를 확인했다
