# UI/UX 기획 구체화 (UX Spec)

막연한 아이디어를 **화면 단위로 바로 구현할 수 있는 UI/UX 설계 문서**로 만드는 것이 목표다.
기능이 "무엇을 하는가"는 기능 기획(/feature-spec)의 몫이고, 이 스킬은 **"사용자가 어떻게 보고, 어떻게 움직이는가"**를 결정한다.

## 절대 원칙

1. **구현(코드)부터 시작하지 않는다.** 문서가 완성되고 사용자가 확인하기 전까지 UI 코드를 쓰지 않는다.
2. **질문은 AskUserQuestion 도구로 한다.** 라운드당 최대 4개. 반드시 구체적 선택지를 제시하고 추천안에 "(추천)"을 붙인다. 레이아웃·플로우처럼 눈으로 비교해야 하는 선택지는 `preview` 필드에 ASCII 목업을 넣어 비교시킨다.
3. **한 단계의 결정이 끝나야 다음 단계로 넘어간다.**
4. **먼저 조사하고 나서 묻는다.** `docs/planning/`의 기능 기획 문서, 기존 화면 코드, 기존 디자인 토큰(색/폰트/컴포넌트)을 먼저 읽고, 이미 정해진 것은 질문하지 않는다. 기능 기획 문서가 아예 없고 기능 자체가 불명확하면 `/feature-spec`을 먼저 진행하자고 제안한다.
5. **"모르겠다 / 알아서 해줘"** 선택 시 근거 있는 추천안을 채택하고 `[가정]`으로 기록, 마지막에 일괄 재확인한다.
6. **단계가 끝날 때마다 산출물 문서를 즉시 갱신 저장한다.**
7. **새 UI 라이브러리·폰트·아이콘 도입은 반드시 사용자 승인을 받는다.** 기존 프로젝트의 디자인 자산을 우선 재사용한다.

## 진행 단계

### 0단계 — 착수

- 기능 기획 문서와 기존 화면/스타일 코드를 읽고, 확정된 사실(타깃, 시나리오, 범위)을 요약한다.
- 산출물 파일 생성: `docs/planning/<기능명>-uiux기획.md`
- 몇 단계로 진행할지 사용자에게 안내한다.

### 1단계 — 사용 맥락

결정할 것: **어떤 기기에서, 어떤 상황의 사용자가** 쓰는가.

질문 예시:
- 주 사용 기기와 우선순위: 모바일 우선? 데스크톱 우선? (반응형 범위)
- 사용 상황: 이동 중 한 손 조작? 책상에서 집중? 남에게 공유받아 처음 진입?
- 사용자의 숙련도와 첫인상: 처음 온 사람이 설명 없이 이해해야 하는 화면인가?
- 톤: 가볍고 캐주얼 vs 신뢰감 있고 차분함 (서비스 성격 기준 후보 제시)

### 2단계 — 정보 구조 (IA)

결정할 것: **화면 목록과 이동 구조**.

- 필요한 화면 전체 목록을 도출해 확인받는다 (화면명 / 목적 / 진입 경로).
- 내비게이션 구조를 선택지로 제시한다 (탭바, 헤더 메뉴, 단일 플로우 등 — ASCII preview로 비교).
- 각 화면의 URL/딥링크 필요 여부 (공유되는 화면인지)를 확인한다.

### 3단계 — 핵심 플로우

결정할 것: 주요 과업별 **화면 흐름과 분기**.

- 기능 기획의 시나리오별로 화면 흐름도를 그려 확인받는다 (mermaid flowchart 권장).
- 각 분기점을 질문으로 확정한다: 실패하면 어디로? 뒤로 가면? 이탈 후 재진입하면?
- 단계가 3개 이상인 플로우는 진행 표시(스텝퍼/프로그레스) 방식을 결정한다.
- 최초 진입(온보딩)과 재방문의 차이를 결정한다.

### 4단계 — 화면별 상세 설계

결정할 것: 화면마다 **레이아웃과 모든 상태**.

화면 하나씩 진행한다 (중요한 화면부터):
- **레이아웃**: 영역 구획을 ASCII 와이어프레임 2~3안으로 제시하고 고르게 한다.
- **우선순위**: 이 화면에서 사용자가 가장 먼저 봐야 할 것 하나, 가장 눌러야 할 버튼 하나(Primary CTA)를 확정한다.
- **상태 4종 세트**: 정상 / 비어있음(empty) / 로딩 / 에러 — 각각 무엇을 보여줄지 결정한다. 특히 empty 상태의 안내 문구와 행동 유도를 정한다.
- **핵심 문구**: 제목, CTA 레이블, 에러 메시지의 실제 문안을 정한다 (톤 결정 반영).
- 화면 수가 많으면 핵심 화면만 상세히 하고 나머지는 패턴 재사용으로 처리한다.

### 5단계 — 인터랙션과 피드백

결정할 것: **행동에 대한 반응** 규칙.

- 주요 액션의 피드백 방식: 토스트? 인라인 메시지? 화면 전환? 낙관적 업데이트?
- 확인이 필요한 파괴적 액션(삭제 등)의 확인 UX
- 전환/애니메이션 수준: 없음 / 최소 / 적극 (기존 프로젝트 관례 우선)
- 접근성 기본선: 터치 타깃 크기, 대비, 키보드 조작 필요 여부

### 6단계 — 비주얼 방향과 마무리

- 컬러·타이포·컴포넌트 방향: 기존 디자인 시스템 재사용을 기본으로 하고, 새 방향이 필요할 때만 후보를 제시한다.
- 레퍼런스가 있으면 받는다 (URL, 서비스명).
- 쌓인 `[가정]`을 일괄 확인받는다.
- 문서 요약을 보여주고, 구현 순서(화면 단위)를 제안한다.

## 산출물 문서 템플릿

```markdown
# <기능명> UI/UX 기획

- 작성일: YYYY-MM-DD / 상태: 초안 | 확정
- 연관 문서: <기능명>-기능기획.md

## 1. 사용 맥락
- 기기 우선순위 / 사용 상황 / 톤앤매너

## 2. 정보 구조
- 화면 목록 (화면명 / 목적 / 진입 경로 / URL)
- 내비게이션 구조

## 3. 핵심 플로우
- 과업별 화면 흐름도 (mermaid) + 분기/실패/재진입 처리

## 4. 화면별 상세 설계
화면마다:
- ASCII 와이어프레임 / 시각 우선순위와 Primary CTA
- 상태별 설계 (정상/empty/로딩/에러) / 핵심 문구
- `[가정]` 표시된 항목은 사용자 미확정 사항

## 5. 인터랙션 규칙
- 피드백 방식 / 파괴적 액션 확인 / 애니메이션 수준 / 접근성

## 6. 비주얼 방향
- 컬러/타이포/컴포넌트 방침, 레퍼런스

## 7. 구현 순서 및 오픈 이슈
```

## 완료 기준

- [ ] 모든 화면이 목록에 있고 진입 경로가 정의돼 있다
- [ ] 핵심 플로우에 실패/이탈/재진입 분기가 정의돼 있다
- [ ] 핵심 화면마다 와이어프레임과 상태 4종(정상/empty/로딩/에러) 결정이 있다
- [ ] 확인 안 받은 `[가정]`이 없다
- [ ] 사용자가 최종 문서를 확인했다
