# 축하 롤링페이퍼 기능 기획

- 작성일: 2026-07-06 / 상태: **확정** (모든 가정 사용자 승인 완료)
- 연관 문서: (없음 — 확정 후 필요 시 /ux-spec 진행)

## 배경

선물모아는 현재 "주최자가 펀딩을 만들고 → 친구들이 참여하고 → 마감"에서 흐름이 끝난다.
두 가지 공백이 있다:

1. 참여자는 한 번 돈을 내고 나면 다시 방문할 이유가 없다.
2. 선물 받는 순간의 "마음 전달"이 카톡방에 흩어져 사라진다.

축하 롤링페이퍼는 참여자들의 축하 메시지를 한곳에 모아 실시간으로 쌓이는 디지털
카드로 만들고, 마감 시 완성본을 주인공에게 전달한다.

> **[방향 확정]** 이번 MVP에서 펀딩은 **주인공이 직접 개설한다 (주최자 = 주인공)**.
> 주인공이 위시리스트를 만들고 제3자가 대신 주최하는 구조는 이번에 만들지 않는다.

### 기획 시작 전 이미 결정돼 있던 사항 (기존 코드/기획에서 확인)

- 참여자는 비로그인으로 이름 + 금액 + 축하 메시지(선택)를 남긴다.
- 참여 금액은 주최자에게만 공개, 축하 메시지는 공개.
- 결제는 목업. 새 라이브러리/외부 서비스 도입은 사용자 승인 필수.
- 펀딩에는 마감일(deadline)이 있고, 이후 참여 불가.

## 1. 문제 정의

- **타깃 사용자**
  - 1차: 선물을 받는 **주인공(=펀딩 개설자)** — 메시지가 쌓이는 과정 자체가 감동 경험.
  - 2차: 축하 메시지를 남긴 **참여자** — 한 번 참여 후 재방문 이유가 없다.
- **해결하는 문제**: 여럿이 돈을 모아 선물해도 "마음"은 카톡방에 흩어져 사라진다.
  선물과 함께 마음을 한곳에 모아 전달하는 장치가 없다.
- **현재의 대안**: 카톡 단체방에서 각자 축하 인사, 또는 종이 롤링페이퍼(오프라인 한정).
- **핵심 목표 (확정)**: **주인공의 감동 경험**이 최우선. 카드 오프닝, 메시지 하나씩 공개
  같은 "받는 순간의 연출"에 투자하고, SNS 공유는 부산물로 따라오게 한다.
- **성공 기준 (확정)**: **주인공 열람율** — 전달된 롤링페이퍼를 주인공이 실제로 열어본 비율.

## 2. 사용자 시나리오 (확정)

**주 시나리오**

1. **주인공·개설**: 주인공이 / 생일 등 기념일을 앞두고 / 갖고 싶은 선물로 펀딩을 만들어 / 친구들에게 공유한다.
2. **참여자·작성**: 친구가 / 링크를 받고 / 참여하며 축하 메시지를 남겨 / 롤링페이퍼에 **실시간으로** 칸이 추가된다.
3. **주인공·감상**: 주인공이 / 펀딩 진행 중 수시로 / 쌓여가는 롤링페이퍼를 본다. / 마감 후 완성본을 본인 계정(/me·펀딩 페이지)에서 감상한다.

**부 시나리오**

4. **참여자·재방문**: 메시지를 남긴 참여자가 / 마감 후 / 완성된 롤링페이퍼를 다시 보러 온다.

**핵심 결정**

- 롤링페이퍼는 마감 전에도 **작성 즉시 공개** (서프라이즈 게이팅 없음).
- 메시지 소스는 **기존 참여 축하 메시지 그대로** — 별도 재작성 없음. 참여(입금) 없이 메시지만 남기는 것은 불가.
- ~~마감 시 완성본을 주인공 가입 이메일로 발송~~ → **5단계에서 보류 결정** (오픈 이슈 참고).

## 3. 범위 (확정)

| 분류 | 기능 |
|---|---|
| **Must** | A. 실시간 롤링페이퍼 뷰 (메시지가 카드로 쌓이는 화면) |
| **Must** | B. 참여 폼 메시지 UX 강화 ("롤링페이퍼에 실려요" 안내·작성 유도) |
| **Must** | C. 오프닝 연출 (카드가 하나씩 펼쳐지는 애니메이션) |
| **Should** | E. 카드 꾸미기 (참여자가 카드 색상/이모지 선택) |
| **Could** | F. 완성본 이미지 저장·SNS 공유 |
| **Won't** | D. 완성본 이메일 발송 — 외부 발송 수단 도입 보류 (오픈 이슈) |
| **Won't** | 제3자 주최 / 주인공 위시리스트 기반 대리 개최 |
| **Won't** | 참여(입금) 없이 메시지만 남기기 |
| **Won't** | 서프라이즈 게이팅 (마감 전 비공개) — 실시간 공개로 확정 |
| **Won't** | G. 열람 추적/측정 로깅 → 성공 기준 측정 문제는 오픈 이슈 참고 |

**MVP 검증**: Must(A+B+C)만으로 주 시나리오 1→2→3이 끝까지 동작한다.
(마감 후 완성본은 이메일 없이도 주인공이 본인 계정/링크로 접근 가능)

## 4. 상세 정책

### A. 실시간 롤링페이퍼 뷰 (Must)

- **위치**: 펀딩 페이지(`/f/[id]`) 내 섹션(미리보기) + 전용 페이지 `/f/[id]/paper`.
  이메일 링크·공유는 전용 페이지 URL을 쓴다.
- **공개 범위**: 펀딩 링크를 아는 누구나 열람 가능(기존 메시지 공개 정책과 동일).
  참여 금액은 기존 정책대로 계속 주인공에게만 공개 — 롤링페이퍼에는 절대 노출하지 않는다.
- **정렬**: 최신순 (새 카드가 맨 위).
- **메시지 없는 참여**: "OO님이 마음을 보탰어요" 식의 이름만 있는 작은 카드로 표시.
- **상태와 전이**: `진행 중`(카드 실시간 추가) → `완성`(마감 시각 도래, 카드 추가 불가).
  전이는 기존 펀딩 마감(deadline)과 동일 시점이며 별도 상태 컬럼이 필요 없다.
- **숨김(주인공 권한)**: 주인공이 카드 숨김/해제 가능(soft). 참여 기록과 금액 집계는 유지.
  - 숨긴 카드는 주인공 포함 모든 방문자 화면에서 비표시(주인공 관리 화면에서만 "숨긴 카드 N장"으로 확인·해제).
- 참여 0건으로 마감 시 "아직 카드가 없어요" 빈 화면.

### B. 참여 폼 메시지 UX 강화 (Must)

- 메시지는 지금처럼 **선택 입력, 최대 200자** 유지.
- 입력란에 "이 메시지는 OO님의 롤링페이퍼에 실려요" 안내로 작성 동기 부여.
- **수정·삭제**: 작성한 **같은 브라우저에서만** 가능(작성 시 발급한 로컬 토큰 기준).
  - 수정·삭제는 마감 전까지만. 마감 후 완성본은 불변.
  - "삭제"는 메시지만 비우고 참여(금액) 기록은 유지 → 이름 카드로 전환.

### C. 오프닝 연출 (Must)

- **마감 후** 전용 페이지 **첫 진입 시 1회** 카드가 하나씩 펼쳐지는 풀 연출. 스킵 버튼 필수.
- 연출 종료 후에는 "처음부터 다시 보기" 버튼으로 재생 가능.
- 진행 중(마감 전)에는 연출 없이 카드 그리드가 실시간으로 보인다.
- "첫 진입" 판별은 방문자 브라우저 로컬 저장 기준(방문자마다 각자 첫 1회).

### D. 완성본 이메일 발송 — **이번 버전 보류**

- 외부 발송 수단(Resend/SMTP) 도입과 마감 감지 트리거 모두 보류. 오픈 이슈로 이관.
- 이번 버전에서는 주인공이 /me와 펀딩 페이지의 "완성된 롤링페이퍼 보기" 동선으로 접근한다.

### E. 카드 꾸미기 (Should)

- 참여 시 프리셋 카드 색상 4~6종 중 택1 + 이모지 1개 선택. 이미지 업로드는 없음.

### F. 이미지 저장·SNS 공유 (Could)

- 완성본을 이미지 1장으로 렌더링해 저장/공유. 금액 정보는 절대 포함하지 않는다.

### 권한 요약

| 행위 | 누가 |
|---|---|
| 롤링페이퍼 열람 | 링크 소지자 전원 (숨긴 카드 제외) |
| 카드 숨김/해제 | 주인공(=펀딩 개설자) |
| 메시지 수정·삭제 | 작성한 브라우저, 마감 전까지 |
| 완성본 이메일 수신 | 주인공 |

### 엣지케이스

- 마감 직전 참여가 마감 처리와 겹치면: 참여 성공 여부는 기존 참여 마감 로직을 따르고, 성공한 참여는 완성본에 포함된다.
- 펀딩 삭제 시 롤링페이퍼(=Contribution)도 기존 cascade로 함께 삭제된다.
- 숨김 상태 카드를 작성자가 같은 브라우저에서 수정하면: 수정은 되지만 숨김 상태는 유지.

## 5. 데이터 및 시스템

### 엔티티

새 엔티티 없음. **롤링페이퍼 카드 = Contribution 레코드 그 자체**이며, 기존
`Funding 1—N Contribution` 관계를 그대로 쓴다. 필드만 확장한다:

| 테이블 | 추가 필드 | 용도 |
|---|---|---|
| `Contribution` | `hiddenAt DateTime?` | 주인공의 카드 숨김 (null = 표시) |
| `Contribution` | `editToken String?` | 작성 브라우저 수정·삭제 인증 (추측 불가 랜덤 값, 응답에 1회만 노출) |
| `Contribution` | `cardColor String?`, `emoji String?` | 카드 꾸미기 (E) |

- 이메일 발송이 보류되어 `Funding.paperEmailSentAt` 등의 필드도 이번에는 추가하지 않는다.

### 시스템

- **실시간 반영**: 기존 폴링 구조(`/api/fundings/[id]/updates` + 가시 상태 `setInterval`)를
  재사용한다. updates 응답에 카드(메시지·꾸미기·숨김 제외) 데이터를 포함시킨다.
  폴링 주기는 기존 값 그대로.
- **라우트**: 전용 페이지 `/f/[id]/paper` 추가. 마감 여부에 따라 진행 중 뷰/완성본(연출) 뷰 분기.
- **외부 연동**: 이번 버전 없음 (이메일 보류, 결제는 기존 목업 유지).
- **비기능**: 지인 단위 소규모 트래픽 전제. 특별한 성능 요구 없음.
  editToken은 서버 검증 필수(클라이언트 신뢰 금지), 금액은 어떤 롤링페이퍼 응답에도 포함하지 않는다.

## 6. 마일스톤

1. **M1 — 데이터·폼**: Contribution 필드 확장, 참여 폼 안내 문구(B), editToken 발급과
   같은 브라우저 수정·삭제. 완료 기준: 참여→메시지 수정→삭제가 동작.
2. **M2 — 롤링페이퍼 뷰(A)**: 펀딩 페이지 섹션 + `/f/[id]/paper` 전용 페이지, 실시간 폴링
   반영, 이름 카드, 최신순, 주인공 숨김/해제. 완료 기준: 두 브라우저로 실시간 확인.
3. **M3 — 오프닝 연출(C)**: 마감 후 첫 진입 풀 연출 + 스킵 + 다시 보기. 완료 기준: 마감된
   펀딩에서 연출 동작. **← 여기까지가 Must, 출시 가능 지점**
4. **M4 — 카드 꾸미기(E)**: 프리셋 색상 + 이모지.
5. **M5 — 이미지 저장·공유(F)**.

## 7. 오픈 이슈

| 이슈 | 내용 | 결정 시점 |
|---|---|---|
| 완성본 이메일 발송(D) | 발송 수단(Resend 등)과 마감 감지 트리거(크론 vs 지연 발송) 모두 미정 | 차기 버전 기획 시 |
| 성공 기준 측정 | 성공 기준은 "주인공 열람율"인데 열람 추적(G)을 이번에 만들지 않아 측정 수단이 없음. 측정을 붙이거나 성공 기준을 정성 평가로 조정 필요 | 출시 전 |
